# Фильтр Блума

CLI-утилита на Go для демонстрации точной и приблизительной дедупликации событий по `event_hash`.  
Проект показывает разницу между точной дедупликацией через `map` и вероятностной дедупликацией через фильтр Блума, включая оценку ложных срабатываний, памяти и скорости.

## Назначение проекта
Утилита решает задачу по обработке потока событий в формате JSONL.  
Основная цель - сравнить:
- точную дедупликацию через `map`;
- приблизительную дедупликацию через Bloom Filter;
- стоимость этих подходов по памяти и производительности.

## Возможности
Проект поддерживает следующие сценарии:
- генерация тестового потока событий;
- чтение входного JSONL-файла построчно;
- точная дедупликация по `event_hash`;
- подготовка результата дедупликации;
- формирование отчёта по обработке;
- запуск контрольного demo-набора через `make demo`;
- запуск unit-тестов через `make test`;
- запуск benchmark-тестов через `make bench`.

## CLI-команды

### Генерация тестового набора
Пример команды генерации событий:
```powershell
.\cmd\bloom-dedup-demo\bloom-dedup-demo.exe generate --count 1000000 --duplicate-ratio 0.05 --out ids.jsonl --seed 42 --sources 10
```

Параметры генератора:
- `--count` - число записей, целое число больше 0;
- `--duplicate-ratio` - доля дублей, от 0 до 0.9;
- `--out` - путь к выходному файлу;
- `--seed` - seed генератора;
- `--sources` - число источников, от 1 до 100.

### Запуск дедупликации
Пример команды для дедупликации и сохранения отчёта:
```powershell
.\cmd\bloom-dedup-demo\bloom-dedup-demo.exe run --in ids.jsonl --config bloom.json --out result.jsonl --report report.json
```

Команда:
- читает входной JSONL-файл;
- выполняет дедупликацию по `event_hash`;
- сохраняет выходной набор;
- формирует отчёт по обработке.

### Benchmark
Пример команды для проведения benchmark тестирования:
```powershell
.\cmd\bloom-dedup-demo\bloom-dedup-demo.exe bench --in ids.jsonl --config bloom.json
```

Команда предназначена для benchmark-сценария и должна использоваться вместе с тестами производительности проекта.

## Формат входных данных

Основной входной файл - JSONL, одна JSON-запись на строку.  
Пример строки:

```json
{"seq":1,"event_id":"evt_000001","event_hash":"b9c950640e1b3740","source":"collector_01","timestamp":"2026-06-16T10:15:00Z"}
```

Описание полей:

- `seq` - порядковый номер записи;
- `event_id` - логический идентификатор события;
- `event_hash` - ключ дедупликации;
- `source` - источник события;
- `timestamp` - время события в формате RFC3339.

## Формат конфигурации

Конфигурация фильтра Блума задаётся JSON-файлом.  
Пример:

```json
{
  "expected_items": 1000000,
  "false_positive_rate": 0.01,
  "hash_family": "fnv64_double_hashing",
  "mode": "bloom"
}
```

Поля конфигурации:

- `expected_items` - ожидаемое число уникальных элементов;
- `false_positive_rate` - желаемая вероятность ложного срабатывания;
- `hash_family` - способ получения хеш-функций;
- `mode` - режим работы фильтра.

## Формат выходных данных

Результат дедупликации сохраняется в отдельный файл.  
Отчёт формируется в JSON, также поддерживается Markdown-формат.

Пример полей отчёта:

- `total_records`;
- `exact_unique`;
- `exact_duplicates`;
- `bloom_new`;
- `bloom_may_duplicate`;
- `estimated_false_positives`;
- `real_false_positive_rate`;
- `bloom_memory_bytes`;
- `exact_map_memory_bytes`;
- `duration_ms`;
- `by_source`;
- `invalid_sources`.

## Алгоритм

Текущая реализация строится вокруг двух подходов:

1. **Точная дедупликация** через `map`, которая позволяет точно определить число уникальных записей и дублей.
2. **Приблизительная дедупликация** через фильтр Блума, который экономит память, но может давать ложные срабатывания.

Фильтр Блума использует:
- битовый массив на `[]uint64`;
- операции установки и проверки битов;
- расчёт параметров `m` и `k` из `expected_items` и `false_positive_rate`.

Фильтр Блума может сказать точно, нет ли элемента в списке, но сказать точно ,что он там есть, он не может. 
Основной частью фильтра является хеширование его уникальных данных в число (индекс битового массива), и так как массив ограничен, а входных элементов может быть много, то могут возникать коллизии.
То есть на несколько разных элементов хеш даст один индекс. Поэтому если элемент помечен "1" (присутствием), то это не значит, что эта единица предназначалась именно **этому** элементу, а не его "коллизии".
Точной информацией будет, если элемент помечен "0" (отсутствием), что означает, что ни этот, ни любой другой элемент с данным индексом ещё не попадался. Именно поэтому бывают ложные срабатывания.

Почему фильтр Блума экономит память?
map хранит полные хеши (16-32 байта каждый) + служебную информацию;
Фильтр Блума хранит только битовый массив фиксированного размера;

## Известные ограничения

Текущая версия проекта имеет следующие ограничения:

- основная дедупликация выполняется глобально по `event_hash`;
- при превышении expected_items реальная вероятность ложных срабатываний растёт;
- максимальный размер входного файла ограничен доступной памятью (читается построчно);
- в быстром режиме без точного сравнения некоторые метрики ложных срабатываний могут быть недоступны.
