# Решение: дедупликация через фильтр Блума

## Постановка задачи

Цель проекта - реализовать CLI-утилиту `bloom-dedup-demo` на Go для дедупликации событий по полю `event_hash` и показать разницу между двумя подходами: точной дедупликацией через `map` и приблизительной дедупликацией через фильтр Блума.

Решение должно:
- генерировать поток событий с заданной долей дублей;
- выполнять точную дедупликацию;
- выполнять приблизительную дедупликацию через Bloom Filter;
- измерять ложные срабатывания;
- сравнивать память и скорость;
- формировать отчёт по результатам обработки.

## Выбранный алгоритм

В проекте используются два подхода к дедупликации: точный через `map` и приблизительный через фильтр Блума.
### 1. Точная дедупликация через `map`

Для каждого события используется ключ `event_hash`.  
Если значение ещё не встречалось, оно добавляется в `map`; если уже встречалось, событие считается дублем.

Преимущество этого подхода - точность: число уникальных записей и дублей определяется без ошибок.  
Недостаток - память растёт пропорционально числу уникальных элементов, потому что `map` хранит полные ключи и служебные структуры.

### 2. Приблизительная дедупликация через фильтр Блума

Фильтр Блума представляет собой битовый массив, реализованный на `[]uint64`, и набор хеш-функций.  
При добавлении элемента вычисляются `k` позиций, и соответствующие биты устанавливаются в `1`. При проверке элемента:
- если хотя бы один бит равен `0`, элемент точно новый;
- если все биты равны `1`, элемент считается возможным дублем.

Этот подход экономит память, так как хранится только битовый массив фиксированного размера, но за это приходится платить вероятностью ложного срабатывания: новый элемент может быть ошибочно признан дублем.

### 3. Расчёт параметров фильтра

Из конфигурации берутся:
- `expected_items`;
- `false_positive_rate`;
- `hash_family`;
- `mode`.

На их основе автоматически рассчитываются:
- `m` - размер битового массива;
- `k` - число хеш-функций.

Это позволяет не задавать параметры фильтра вручную и подбирать их под ожидаемую нагрузку.

В основе расчёта используются стандартные формулы для Bloom Filter:

m = ceil( -n * ln(p) / (ln(2)^2) )

k = round( (m * ln(2)) / n )

где:
- `n` - ожидаемое число элементов;
- `p` - допустимая вероятность ложного срабатывания;
- `m` - размер битового массива;
- `k` - число хеш-функций.

Чем меньше допустимая вероятность ложного срабатывания, тем больше требуется памяти под фильтр.  
При увеличении ожидаемого числа элементов размер фильтра также растёт.

## Компромиссы решения

Главная идея проекта - показать компромисс между точностью и экономией памяти.

Точная дедупликация через `map`:
- даёт корректный результат без вероятностной ошибки;
- удобна как эталон для сравнения;
- требует больше памяти при росте числа уникальных событий.

Фильтр Блума:
- использует существенно меньше памяти;
- работает быстро за счёт битового массива и простых операций;
- допускает ложные срабатывания, поэтому не может считаться абсолютно точным.

`map` используется как контрольный инструмент для измерения реальной доли ложных срабатываний.

## Обработка входных данных

Входной файл читается построчно в формате JSONL.

Для каждой строки обрабатываются поля:
- `seq`;
- `event_id`;
- `event_hash`;
- `source`;
- `timestamp`.

Поле `source` не участвует в основном ключе дедупликации, но используется как дополнительное измерение в отчёте.  
Статистика ведётся глобально по `event_hash`, а также по отдельным источникам, если поле `source` присутствует и прошло валидацию.

Некорректные строки, невалидные источники и отсутствующие необязательные поля не должны приводить к аварийному завершению программы.  
Ошибки не должны останавливать обработку всего файла. Битые строки и невалидные источники учитываются отдельно в отчёте.

## Контрольный запуск

Для быстрой проверки проекта используется фиксированный контрольный набор из каталога `testdata/control`.

Запуск выполняется командой:

```powershell
make demo
```

В результате формируются:
- `output/demo_result.jsonl`;
- `output/demo_report.md`.

На контрольном наборе из 30 записей получен следующий результат:

- `total_records`: 30
- `bad_lines`: 0
- `exact_unique`: 22
- `exact_duplicates`: 8
- `bloom_new`: 21
- `bloom_may_duplicate`: 9
- `estimated_false_positives`: 1
- `real_false_positive_rate`: 0.045454545454545456
- `bloom_memory_bytes`: 36
- `exact_map_memory_bytes`: 6984

Статистика по источникам:
- `collector_01`: 13 записей, 11 уникальных, 2 дубля;
- `collector_02`: 6 записей, 5 уникальных, 1 дубль;
- `collector_03`: 11 записей, 10 уникальных, 1 дубль.

Некорректные источники в контрольном наборе отсутствовали: `invalid_sources = []`.

По результатам demo видно, что фильтр Блума действительно может давать ложные срабатывания: из 22 реально уникальных элементов один был ошибочно посчитан возможным дублем.

## Benchmark

Для сравнительного измерения скорости и памяти реализована отдельная команда `bench`.

Запуск выполняется командой:

```powershell
make bench
```

или напрямую:

```powershell
.\cmd\bloom-dedup-demo\bloom-dedup-demo.exe bench --in .\testdata\control\demo_events.jsonl --config .\testdata\control\demo_config.json
```

Команда `bench` выводит в консоль сравнительную характеристику двух подходов:
- число уникальных записей;
- число дублей;
- число ложных срабатываний;
- оценку вероятности ложного срабатывания;
- потребление памяти;
- время выполнения;
- пропускную способность в строках в секунду.

На малом контрольном наборе benchmark по времени малоинформативен, потому что обработка занимает менее 1 мс, поэтому в выводе могут появляться нулевые значения времени и производительности.

На больших наборах данных benchmark показывает более показательную разницу между точной дедупликацией и Bloom Filter.  
При этом результаты времени и памяти могут немного отличаться между запусками из-за особенностей машины, среды выполнения Go, фоновой нагрузки, работы сборщика мусора и состояния файлового кэша.

Эти колебания не считаются ошибкой.  
Их следует трактовать как естественную особенность benchmark-измерений.

## Известные ограничения

Текущая реализация имеет несколько ограничений.

1. Основная дедупликация выполняется глобально по `event_hash`.  
   Поле `source` участвует только в отчётности и не влияет на основной ключ дедупликации.

2. Реальная вероятность ложных срабатываний зависит от корректности выбора `expected_items` и `false_positive_rate`.  
   При превышении ожидаемого числа элементов фактическая ошибка будет расти.

3. На больших наборах данных точность Bloom Filter снижается не из-за ошибки реализации, а из-за природы вероятностного алгоритма.  
   Чем плотнее заполняется битовый массив, тем выше вероятность того, что новый элемент будет ошибочно признан возможным дублем.

4. В режиме точного сравнения дополнительная память расходуется на `map`, поэтому такой режим удобен для анализа и тестирования, но хуже масштабируется.

5. На малых наборах данных benchmark по времени и пропускной способности малоинформативен, так как измеряемое время может быть меньше 1 мс.

## Вывод

В проекте получился рабочий пример сравнения точной и приблизительной дедупликации.  
Точный вариант через `map` удобно использовать как эталон, а фильтр Блума показывает, как можно заметно сэкономить память ценой возможных ложных срабатываний. Контрольный запуск, тесты и benchmark подтверждают, что базовая логика работает корректно.
